Product Requirements Document: Forensic-Hawkeye-Env
1. Executive Summary
Concept: A mathematically deterministic, 2.5D accident reconstruction environment for AI agents.
Objective: The AI acts as a forensic liability auditor. It must iteratively tune physics simulation parameters (speed, steering, timing) to perfectly recreate the final resting state of an accident, thereby proving or disproving human testimony.
Hackathon Win Condition: Flawless compliance with the OpenEnv Pydantic specification while executing hidden pymunk rigid-body physics under the hood to ensure a deterministic, un-hallucinatable grader.
2. Core Architecture & Constraints
* Framework: OpenEnv (Python 3.10+).
* Physics Backend: pymunk (2D physics engine) executing headless.
* Math Hack (The 2.5D Illusion): The LLM operates in X/Y coordinates. The backend calculates Z-axis longitudinal weight transfer for braking to alter the dynamic friction coefficient (  ), ensuring physics realism without 3D computational overhead.
* Hardware Limit: Must run cleanly on 2 vCPUs, 8GB RAM. Do not include GUI libraries (e.g., pygame, matplotlib) in the main execution loop.
3. Pydantic API Models (OpenEnv Interface)
The LLM does not see the physics engine. It only sees this data contract.
Observation Space
class Observation(BaseModel):
   target_debris: Dict[str, Tuple[float, float]] # e.g., {"Car_A": (45.2, 12.1)}
   simulated_debris: Dict[str, Tuple[float, float]] # Results from the last test
   distance_errors: Dict[str, float] # Delta between target and simulation
   human_testimony: str # The lie the agent must disprove
   active_contradiction_flag: bool # True if physics disproves testimony

Action Space
class Action(BaseModel):
   action_type: Literal["RUN_SIMULATION", "SUBMIT_VERDICT"]
   # Used for RUN_SIMULATION
   sim_parameters: Optional[Dict[str, Dict[str, float]]] # e.g. {"Car_A": {"speed": 45.0, "steering": 0.0}}
   # Used for SUBMIT_VERDICT
   liable_party: Optional[str]
   root_cause: Optional[str]

4. The 3-Task Progression & Grader Logic
Task 1 (Easy): The Public Property Strike
   * Scenario: Car A hits a bus stop. Driver claims they drove the 30mph speed limit.
   * Constants: Bus stop (X,Y). Car A Mass.
   * Variables: Car_A.speed, Car_A.steering.
   * Mechanics: Agent iterates RUN_SIMULATION. Distance error drops as agent increases speed.
   * Grader: * 1.0: Agent's last simulation has distance_error < 1.0m AND SUBMIT_VERDICT blames Car A for Speeding (true speed = 54mph).
   * 0.0: Agent submits verdict before lowering error   , or believes the 30mph lie.
Task 2 (Medium): The Temporal Pedestrian Paradox
      * Scenario: Car A swerves into a pole. Claims a pedestrian ran into the road. Pedestrian claims they were walking slowly.
      * Constants: Pole (X,Y), Pedestrian Origin (X,Y).
      * Variables: Car_A.speed, Car_A.steering, Pedestrian.velocity.
      * Mechanics: Agent matches the pole crash, then runs a counterfactual simulation (Car_A.steering = 0). Calculates if the straight-line trajectory intersects the pedestrian's path based on different pedestrian speeds.
      * Grader:
      * 1.0: Agent runs the counterfactual, proves that a slow-walking pedestrian would not have been hit, and blames the driver for an overreaction.
Task 3 (Hard): The Multi-Vehicle Origin (Conservation of Momentum)
      * Scenario: 3-car intersection pileup. All cars scattered. Nobody admits to running the red light.
      * Constants: Masses of Cars A, B, C. Target debris coordinates for all three.
      * Variables: speed and timing_offset for Cars A, B, and C (6 variables total).
      * Mechanics: Because of   , there is only one valid set of entry speeds that results in the specific final scatter plot. Agent must optimize all 6 variables to minimize combined distance error.
      * Grader:
         * 1.0: Agent successfully tunes the engine to < 1.5m combined error, revealing Car B was going 85mph. Submits verdict blaming Car B.
5. Reward Shaping (Continuous Signal)
Do not use sparse (binary) rewards.
         * +0.1: Executing a valid RUN_SIMULATION.
         * +0.2: Reducing the distance_error compared to the previous step.
         * -0.1: Increasing the distance_error (moving away from the solution).
         * -0.5: Calling SUBMIT_VERDICT when distance_error is still   .
         * +0.5: Correct final verdict submission.
